**Owner:** TBD
**Status:** Plan — not yet started

---

## Goal

The `spec-compliance-watchdog` agent runs at QA Gate 6 (after manual verification, before commit) and detects semantic drift between story acceptance criteria, test-design scenarios, and the actual implementation. Tier 3 already covers the happy path manually (`TEST-GUIDE.md` §23). What is missing is:

- A canonical agent description in `HOW-IT-WORKS.md` that downstream docs can cite.
- Drift-scenario coverage in Tier 3 beyond the single happy-path test.
- Reproducible drift inputs in `TEST-INPUTS.md`.
- Strategy updates so Tier 1 (structural) and Tier 2 (log-replay) tests are formally part of the pyramid.
- Implementation of those Tier 1 and Tier 2 tests.

Two stages in order: **docs first, tests second.** Within each, files are sequenced so each one consumes the previous one's outputs.

## Sequencing rules

- **Stage 1 must finish before Stage 2 starts.** Strategy drives implementation; if strategy is wrong, tests will be wrong.
- **Phase 0 gates everything.** If no baseline story exists, Phase 7 changes shape (synthetic story authored in TEST-INPUTS instead of copied).
- **No edits to `.claude/agents/spec-compliance-watchdog.md` itself.** All work is in `QA-TESTS/` and a small read of the existing agent file. The agent under test stays unmodified.
- **Each phase ends with a verification step:** Phase 1 ends with HOW-IT-WORKS render check; Phase 2 ends with TEST-GUIDE TOC consistency; Phase 4 ends with grep for stale tier references; Phases 5–6 end with `npm test` in `QA-TESTS/`.
